# Project Structure Report

## Objective

Reorganize the Punjab DSR project into a professional two-folder application structure:

```text
frontend/   Next.js web application and legacy portal assets
backend/    Express API, Prisma schema, worker and backend services
```

The previous structure used nested workspace folders:

```text
apps/web
apps/api
packages/shared
```

The new structure is easier to understand for Vercel, Render, future developers and non-technical project reviewers.

## Final Structure

```text
punjab_dsr/
  backend/
    deployment/
    docs/
    prisma/
    reports/
    scripts/
    src/
    Dockerfile
    package.json
    package-lock.json
    tsconfig.json

  frontend/
    app/
    components/
    lib/
    public/
    Dockerfile
    next.config.ts
    package.json
    package-lock.json
    tsconfig.json
    vercel.json
```

## Files and Folders Moved

| Old Path | New Path | Reason |
|---|---|---|
| `apps/api` | `backend` | Clear backend service ownership and Render deployment root. |
| `apps/web` | `frontend` | Clear frontend service ownership and Vercel deployment root. |

## Files and Folders Removed

| Removed Path | Reason |
|---|---|
| `packages/shared` | It was not imported by the current source code. Removing it keeps the requested two-folder structure clean. |
| `apps/` | No longer needed after moving frontend and backend source folders. |
| `apps/web/.next` | Generated Next.js build output; ignored and regenerated by build. |
| `apps/web/node_modules` | Generated dependencies; ignored and regenerated by `npm install`. |

## Configuration Updates

| File | Change |
|---|---|
| `backend/package.json` | Backend is standalone for Render and local API work. |
| `frontend/package.json` | Frontend is standalone for Vercel and local web work. |
| `backend/Dockerfile` | Uses backend-local `package.json` and build/start commands. |
| `frontend/Dockerfile` | Uses frontend-local `package.json` and build/start commands. |
| `backend/deployment` | Contains Docker Compose and environment templates. |
| `backend/docs` | Contains deployment guides, project report and archived docs. |
| `frontend/scripts/maintenance` | Contains frontend-only maintenance scripts. |

## Security Cleanup

| File | Change |
|---|---|
| `backend/src/lib/email.ts` | Removed hardcoded Brevo API key fallback. The backend now requires `BREVO_API_KEY` through environment variables when email sending is used. |

## Frontend Deployment

Vercel settings:

```text
Root Directory: frontend
Install Command: npm install --legacy-peer-deps
Build Command: npm run build
Output Directory: .next
```

Required environment variable:

```text
NEXT_PUBLIC_API_BASE_URL=https://<render-backend-url>.onrender.com
```

## Backend Deployment

Render settings:

```text
Root Directory: backend
Build Command: npm install --legacy-peer-deps && npm run build
Start Command: npm run start
```

Required environment variables:

```text
NODE_ENV=production
PORT=8080
DATABASE_URL=<Neon PostgreSQL URL>
QUEUE_REDIS_URL=<Upstash Redis URL>
REDIS_URL=<Upstash Redis URL>
JWT_SECRET=<long random secret>
JWT_REFRESH_SECRET=<different long random secret>
WEB_ORIGIN=<Vercel frontend URL>
PUBLIC_APP_URL=<Vercel frontend URL>/legacy
LOCAL_FILE_STORAGE=true
```

Optional email variables:

```text
BREVO_API_KEY=<Brevo API key>
SMTP_USER=<verified sender email>
```

## Verification

Run from project root:

```bash
cd backend && npm install --legacy-peer-deps && npm run build
cd ../frontend && npm install --legacy-peer-deps && npm run build
```

The backend and frontend are intentionally built separately.

## Notes

- The repository root is intentionally kept minimal. Supporting docs and scripts live inside `backend` or `frontend`.
- Free hosting remains suitable for demo/testing only. Paid production should move file storage away from Render local disk.
